# 流程描述规范 — 内容质量三模式

技能文档流程描述的详细规范，覆盖流程结构、描述语言和异常处理三个模式。

> 本文件不足 300 行，因此不需要目录。

## 模式4：流程描述规范

技能执行流程的描述本质上是该技能的"内部 SOP"，应清晰回答：**在什么状态下开始 → 经过哪些步骤 → 产生什么结果 → 遇到问题如何处理**。

### 五要素

一个完备的流程描述必须覆盖以下 5 个部分：

1. **触发条件与前置检查** — 明确该技能可以被调用的前提：必填参数、系统状态、权限要求。
   - 示例："调用前必须提供 `order_id`，且该订单属于当前登录用户；若未登录，技能将直接返回权限错误。"

2. **主流程（Happy Path）** — 用有序步骤描述正常处理逻辑。每步以动词开头，说明动作和中间产物。
   - 强调"外部可见行为"，而非内部代码实现。例如："调用物流第三方 API 查询轨迹" 而不是 "调用 `requests.get(url)`"。

3. **分支与决策点** — 在关键节点描述条件判断和对应的不同路径。
   - 使用 `IF...THEN...ELSE` 结构，标注导致分支的变量来源（参数、API 返回码等）。

4. **异常与容错处理** — 列出可能的失败类型（超时、无数据、格式错误等），及对应的恢复动作。
   - 必须明确：自动重试的条件/次数、是否降级返回缓存数据、还是转交给用户决策。

5. **输出规格与副作用** — 成功时的返回数据结构、失败时的错误标准格式。
   - 注明该技能是否修改了外部系统状态（写库、发消息、扣费等），这是智能体规划后续步骤的关键。

### 推荐格式

#### 编号步骤式（最通用）

```text
执行流程：
1. 验证参数：检查 `city` 是否为空，若为空则立即返回错误 `MISSING_PARAM`。
2. 查询天气：调用内部天气接口，传入 `city`，等待响应，超时阈值 3 秒。
3. 判断查询结果：
   a. IF 返回成功且数据有效：进入步骤4。
   b. ELSE IF 返回城市不存在：返回错误 `UNKNOWN_CITY`，附带近似城市名列表。
   c. ELSE IF 超时或服务异常：返回错误 `SERVICE_UNAVAILABLE`，建议稍后重试。
4. 格式化输出：将原始数据提取，按标准 JSON 结构返回。
注意：本技能为只读操作，不会修改任何系统状态。
```

#### 决策树式（适合复杂分支）

将 Mermaid 流程图转化为"决策树式"文字叙述。核心相同：结构化分步描述。

### 反模式

1. **缺少触发条件** — 智能体不知道何时该调用此技能，或在缺少前置条件时盲目调用。
2. **只有 Happy Path** — 智能体在遇到意外情况时"蒙眼狂奔"，不知道如何处理。
3. **模糊步骤** — "然后处理一下数据"没有说明具体动作和预期结果。
4. **暴露实现细节** — 让智能体看到 SQL 查询或内存操作，而非逻辑语义。
5. **未声明副作用** — 智能体不知道该技能修改了外部系统，导致后续规划出错。

## 模式5：描述语言规范

技能文档的核心读者是 LLM，因此流程描述必须"像提示词一样精炼且精准"。

### 五条语言规则

1. **使用确定性语言** — 避免模糊词。对每个步骤的结果给出明确断言。

   | 禁止 | 替代 |
   |------|------|
   | 可能 | 必然 / 若 X 则 Y |
   | 大概 | 具体数值（5 秒、20 条） |
   | 有时候 | 当 X 条件满足时 |
   | 尽量 | 必须 / 应当 |

2. **参数与变量用标识符标注** — 如 `` `order_id` ``，使其与 JSON Schema 中的参数名一致。

3. **时间与数值严格量化** — 说"5 秒超时"而不说"短期超时"；说"返回最多 20 条记录"而不说"返回一些记录"。

4. **以"你"（智能体）为主语** — 智能体以第一视角执行技能。
   - 示例："你收到返回码 `0` 时，表示成功，可继续下一步；收到 `1001` 时，表示参数错误，你需要修正 `city` 参数后重试。"

5. **禁止暴露内部实现细节** — 不让智能体看到数据库 SQL、内存操作、私有函数名，只描述逻辑语义。

### 反模式与修正

#### 反模式：模糊条件

```markdown
# 坏
如果查询失败，可以重试或者提示用户。
```

```markdown
# 好
IF 查询返回超时（>5s），自动重试 1 次（间隔 2s）；仍失败则返回
`API_TIMEOUT`，并建议用户稍后重试或切换至备用数据源技能。
```

#### 反模式：技术实现代替逻辑语义

```markdown
# 坏
调用 `db.execute("SELECT * FROM orders WHERE id = ?")` 获取订单。
```

```markdown
# 好
查询 `order_id` 对应的订单记录。
```

#### 反模式：无量化约束

```markdown
# 坏
短时间内没有响应就重试。
```

```markdown
# 好
若 5 秒内无响应，自动重试 1 次。
```

## 模式6：异常处理规范

智能体的健壮性高度依赖技能的错误信息质量，因此异常流程描述必须自成体系。

### 错误码格式

每个错误码对应一个固定的含义和修复提示，格式：

`ERROR_CODE | 可读消息 | Agent 下一步建议`

示例：
- `MISSING_PARAM | 缺少必填参数 city | 引导用户提供城市名称后重试`
- `API_TIMEOUT | 天气服务超时 | 建议用户稍后重试或切换至 `GetCachedWeather` 技能`
- `ACCOUNT_BANNED | 账号已被封禁 | 终止任务并告知用户联系管理员`

### 两类错误

| 类型 | 特征 | 智能体应做 |
|------|------|-----------|
| **可恢复** | 参数缺失、权限不足、临时超时 | 引导用户补充信息或自动重试 |
| **终结** | 账号封禁、服务永久不可用、数据已删除 | 终止任务并告知用户 |

在流程描述中为每个错误点标注其类型，使智能体知道是继续尝试还是停止。

### 降级路径

如有备用技能或方案，在错误描述中直接建议：

```markdown
IF 主数据源返回 `DATA_EXPIRED`，请尝试调用 `GetCachedReport`
技能获取昨日报表。
```

```markdown
IF 征信查询返回 `SERVICE_UNAVAILABLE`，跳过征信模块，
继续执行其他风险维度检查，在报告中标注"征信数据暂不可用"。
```

### 反模式

1. **无错误码** — 只说"出错了"，智能体无法程序化判断错误类型。
2. **错误码无语义** — `ERR_001` 不如 `MISSING_PARAM` 有指导意义。
3. **不区分可恢复与终结** — 智能体在账号封禁时仍尝试重试，浪费回合。
4. **无降级建议** — 智能体遇到错误后不知道还有什么替代方案。

## 3C 规范总结

三个内容质量模式共同遵循"3C 规范"：

- **Complete（完整）** — Happy Path、分支、异常全覆盖，不让智能体在意外情况下"蒙眼狂奔"。
- **Clear（清晰）** — 使用确定性语言和智能体第一视角，避免歧义和实现细节。
- **Consistent（一致）** — 参数名、状态码、返回格式在整个技能库中保持统一，形成习惯。

## 完整示例

**技能名称**：send_email
**功能**：发送一封带附件的邮件
**触发条件**：必须提供合法的 `to`、`subject`、`body`；`attachments` 可选。

**执行流程**：
1. **参数校验**：
   - IF `to` 地址格式不合法 → 返回 `INVALID_RECIPIENT | 收件人地址无效 | 检查并修正邮箱格式`。
   - IF 附件总大小 > 10MB → 返回 `ATTACHMENT_TOO_LARGE | 附件超出限制 | 提示用户压缩或分段发送`。
2. **连接邮件服务**：
   - 连接 SMTP 服务器，超时 5 秒。
   - 若连接失败，重试 1 次（间隔 2 秒）；仍失败则返回 `SMTP_UNREACHABLE | 邮件服务不可达 | 建议稍后重试`。
3. **发送邮件**：
   - 发送后等待服务器应答。
   - IF 返回成功 → 步骤4。
   - ELSE IF 收件人不存在 → 返回 `MAILBOX_NOT_FOUND | 收件人邮箱不存在 | 告知无法送达`。
   - ELSE IF 其他发送错误 → 返回 `SEND_FAILED | 邮件发送失败 | 携带脱敏错误摘要，建议检查内容后重试`。
4. **记录日志并返回**：
   - 将发送状态写入审计日志（内部操作，智能体无需感知）。
   - 返回成功结果，包含邮件 ID 和发送时间戳。

**副作用**：会在邮件服务商服务器留下发送记录，可能触发用户邮箱通知。
**安全要求**：此技能必须经用户二次确认后才能执行。

## 检查清单

对于每个技能的流程描述：

- [ ] 覆盖五要素：触发条件、主流程、分支决策、异常处理、输出与副作用。
- [ ] 无模糊词（可能、大概、有时候、尽量）。
- [ ] 参数名用反引号标注且与 JSON Schema 一致。
- [ ] 时间与数值已量化。
- [ ] 以"你"为主语描述动作。
- [ ] 无内部实现细节（SQL、内存操作、私有函数名）。
- [ ] 错误码携带语义（`CODE | 消息 | 建议` 格式）。
- [ ] 区分可恢复与终结错误。
- [ ] 提供降级路径或替代方案建议。
