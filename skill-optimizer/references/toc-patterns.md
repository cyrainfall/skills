# 目录 — 大型引用文件的模式

为超过 300 行的引用文件添加目录的详细模式和规则。

> 本文件不足 300 行，因此不需要目录。

## 何时需要目录

当引用文件正文（不含 frontmatter）超过 **300 行**时添加目录。审计脚本会自动标记缺失目录的文件。

300 行以下，目录只会增加噪音而无导航收益 — 跳过。

## 放置位置

目录紧接在 H1（文件标题）之后、任何其他内容之前：

```markdown
# 文件标题

## 目录

- [段落 A](#段落-a)
- [段落 B](#段落-b)

---

## 段落 A
...
```

使用 `---` 分隔目录和正文。替代方案：将目录放在引用块中，但分隔线更清晰。

英文（`## Table of Contents`）和中文（`## 目录`）标题均被审计脚本接受。

## 锚点生成规则

GitHub / 通用 markdown 锚点规则：

1. 将标题转为小写。
2. 去除标点符号（`.`、`,`、`:`、`!`、`?`、`(`、`)`、`` ` `` 等）。
3. 将空格替换为连字符。
4. 保留标题中已有的连字符。
5. 去除首尾连字符。

| 标题 | 锚点 |
|------|------|
| `## Quick Start` | `#quick-start` |
| `## Step 1: Audit` | `#step-1-audit` |
| `## Error Handling (FAQ)` | `#error-handling-faq` |
| `## Using \`pdfplumber\`` | `#using-pdfplumber` |
| `## 数据抽取` | `#数据抽取` |
| `### Sub-step 2.1` | `#sub-step-21` |

当存在重复标题（两个 `## Overview` 段落）时，第二个变为 `#overview-1`，第三个变为 `#overview-2`，以此类推。

## 深度规则

- 包含 `##`（H2）和 `###`（H3）标题。
- 到深度 3 为止。更深的嵌套会使目录杂乱且很少有助于导航。
- H3 条目缩进 2 个空格以显示层次。

## 最简模板

```markdown
# 主题名称

## 目录

- [概览](#概览)
- [主工作流](#主工作流)
  - [步骤1](#步骤1)
  - [步骤2](#步骤2)
- [示例](#示例)
- [故障排除](#故障排除)

---

## 概览
...
```

## 分组段落模板

对于有明确阶段的文件，将条目分组在纯文本标签下（不可链接）：

```markdown
## 目录

**核心概念**
- [本文件覆盖范围](#本文件覆盖范围)
- [术语](#术语)

**流程**
- [初始设置](#初始设置)
- [日常工作流](#日常工作流)
- [关闭](#关闭)

**附录**
- [完整 API 参考](#完整-api-参考)
- [错误码](#错误码)
```

分组标签是加粗文本，不是标题 — 它们本身不出现在目录条目中。

## 嵌套目录示例

对于 H3 子段落内容丰富的情况：

```markdown
## 目录

- [概览](#概览)
- [字段类型](#字段类型)
  - [文本字段](#文本字段)
  - [复选框字段](#复选框字段)
  - [下拉字段](#下拉字段)
  - [签名字段](#签名字段)
- [验证](#验证)
  - [必填检查](#必填检查)
  - [格式检查](#格式检查)
- [错误恢复](#错误恢复)
```

## 反模式

### 反模式：目录条目不可点击

```markdown
# 坏
- 段落一（见下文）
- 段落二（见下文）
```

每个条目必须是 `[标签](#锚点)` 链接。

### 反模式：在目录中包含 H4+

```markdown
# 坏
- [字段类型](#字段类型)
  - [文本字段](#文本字段)
    - [必填文本](#必填文本)
    - [可选文本](#可选文本)
      - [最大长度](#最大长度)
```

到 H3 为止。如果 H4 内容足够重要需要链接，将其提升为 H3。

### 反模式：过时目录

添加或重命名标题时，在同一编辑中更新目录。过时的目录比没有目录更糟 — 它会将读者引导到失效的锚点。

### 反模式：手动猜测锚点

不要凭空发明锚点。机械地应用锚点规则，或从 GitHub 渲染预览中复制。

## 更新步骤

编辑已有目录的文件时：

1. 完成内容编辑。
2. 如果添加、重命名或删除了任何 `##` 或 `###` 标题：更新目录。
3. 验证所有目录锚点仍可解析（每个 `[标签](#锚点)` 指向现有标题）。

## 自动化

目录可以机械地生成。`scripts/check_skill.py` 中的审计脚本不会自动生成目录，但会标记需要目录的文件。手动生成：

1. 按序列出所有 `##` 和 `###` 标题。
2. 对每个标题应用锚点规则。
3. H3 条目缩进 2 个空格。
4. 将结果列表插入 H1 之后紧接的 `## 目录` 下。

## 检查清单

对于每个超过 300 行的引用文件：

- [ ] H1 之后紧接存在 `## Table of Contents`（或 `## 目录`）段落。
- [ ] 正文中每个 `##` 和 `###` 标题都有对应的目录条目。
- [ ] 所有锚点遵循 GitHub 锚点规则。
- [ ] H3 条目缩进 2 个空格。
- [ ] 没有条目指向 H3 以下的标题。
- [ ] 目录与正文用 `---` 分隔。
